# rubric_eval_harness

A small, fast, hackable harness for **iterating on rubric designs** before
scaling them up. Researchers can drop in a new YAML rubric, point at a JSONL of
prompts, and get scores from a small judge model in under a minute.

```
prompts.jsonl  +  rubric.yaml  +  judge model  в”Җв”Җв–¶  scores + summary + plots
```

## Design goals

1. **Fast iteration** вҖ” no service to start, no auth, no GPU required. The
   default `mock` judge produces structurally valid scores so the full pipeline
   runs in seconds.
2. **Pluggable rubric interface** вҖ” three rubric kinds ship out of the box
   (`scalar`, `pairwise`, `multi_criteria`); a new kind is one dataclass + one
   `Scorer`.
3. **Pluggable models** вҖ” `mock`, Hugging Face local, llama.cpp, and any
   OpenAI-compatible endpoint (vLLM / Ollama / OpenAI). All optional extras.
4. **Deterministic mode** for debugging вҖ” fixed seed, `temperature=0`, and the
   mock model is hash-stable per prompt so runs are bit-identical.
5. **Batch *and* streaming** evaluation modes from the same engine.


## 60-second workflow

```bash
# 1. Run the included scalar-helpfulness experiment with the mock judge.
run_experiment --config examples/configs/scalar_helpfulness.yaml

# 2. Inspect outputs.
cat outputs/scalar_helpfulness_demo/summary.txt
cat outputs/scalar_helpfulness_demo/results.jsonl | head -n 2

# 3. Iterate: tweak examples/rubrics/scalar_helpfulness.yaml, rerun.
run_experiment --config examples/configs/scalar_helpfulness.yaml --visualize

# 4. Try the other two rubric kinds.
run_experiment --config examples/configs/pairwise_quality.yaml
run_experiment --config examples/configs/multi_criteria.yaml
```

## CLI

```
run_experiment --config <path>              # primary command
              [--mode batch|stream]
              [--limit N]                   # cap dataset for fast iteration
              [--visualize | --no-visualize]
              [--non-deterministic]         # use config temperature, no seed
              [--seed N]
              [--output-dir DIR]
              [--describe-only]             # print resolved config and exit
```

## Inputs

### Dataset (JSONL)

Each line is one item. Required: `prompt`. Optional: `id`, `response`,
`response_b` (for pairwise), `reference`, `metadata`.

```jsonl
{"id": "q1", "prompt": "What is 2+2?", "response": "4", "response_b": "22"}
```

If `response` is missing and `candidate:` is set in the config, responses are
generated by the candidate model.

### Rubric (YAML or JSON)

All three kinds share `name`, `description`, `judge_prompt_template`,
`metadata`. The template can reference `{prompt}`, `{response}`,
`{reference}`, `{scale_min}`, `{scale_max}`, `{response_a}`, `{response_b}`,
`{label_a}`, `{label_b}`, `{criteria}` depending on kind.

| Kind             | Output type            | Template fields used                                           |
|------------------|------------------------|----------------------------------------------------------------|
| `scalar`         | `float` clamped to range | `prompt`, `response`, `reference`, `scale_min`, `scale_max`    |
| `pairwise`       | `"A" \| "B" \| "TIE"`  | `prompt`, `response_a`, `response_b`, `label_a`, `label_b`     |
| `multi_criteria` | `dict[name -> float]`  | `prompt`, `response`, `reference`, `criteria`                  |

See `examples/rubrics/` for one of each.

### Experiment config

```yaml
name: my_experiment
rubric: ../rubrics/scalar_helpfulness.yaml
dataset: ../data/prompts.jsonl
output_dir: ../../outputs

judge:                # ModelClient config (mock / hf / llama_cpp / openai)
  backend: mock

candidate:            # optional: generate responses on the fly
  backend: hf
  model_id: Qwen/Qwen2.5-0.5B-Instruct

mode: batch           # batch | stream
deterministic: true   # forces seed + temp=0
seed: 0
temperature: 0.0
max_new_tokens: 128
limit: null           # cap dataset size for fast iteration
visualize: false
```

## Outputs

For each run, `outputs/<experiment_name>/` contains:

| File                | Contents                                                       |
|---------------------|----------------------------------------------------------------|
| `results.jsonl`     | One row per item: parsed score, judge text, prompt/response.   |
| `summary.json`      | Aggregate stats keyed for downstream tooling.                  |
| `summary.txt`       | Human-readable table (mean, distribution, per-criterion).      |
| `distribution.png`  | Histogram / bar chart (only when `--visualize` and `[viz]`).   |

## Architecture

```
rubric_harness/
в”ңв”Җв”Җ rubrics/         schema (Scalar/Pairwise/MultiCriteria) + YAML/JSON parser
в”ңв”Җв”Җ models/          ModelClient ABC + mock / hf / llama_cpp / openai backends
в”ңв”Җв”Җ evaluation/      Scorer per rubric kind + batch & streaming engines
в”ңв”Җв”Җ experiments/     Config dataclass + end-to-end runner
в”ңв”Җв”Җ outputs/         JSONL/JSON/text writer + matplotlib visualization
в””в”Җв”Җ cli.py           run_experiment entry point
```

## Determinism

When `deterministic: true` (the default):

- `temperature` is forced to `0.0`
- The configured `seed` is passed to the model on every call
- The mock model's output is a function of `(prompt, seed)` only

With these settings, two runs of the same config produce bit-identical
`results.jsonl`. This is verified by `tests/test_smoke.py`.

## Adding a new rubric kind

1. Add a dataclass to `rubric_harness/rubrics/schema.py` (and to `RubricKind`).
2. Handle it in `rubric_harness/rubrics/parser.py::load_rubric_dict`.
3. Add a `Scorer` subclass in `rubric_harness/evaluation/engine.py` and
   register it in `get_scorer`.
4. (Optional) Add summary / visualization branches in `outputs/`.

## Adding a new model backend

1. Add a `ModelClient` subclass in `rubric_harness/models/<backend>.py`.
2. Register it in `rubric_harness/models/factory.py::build_model`.
3. Add an optional extra in `pyproject.toml` if it brings new dependencies.
